### 总体约束

- **最小修改原则**：只修改重复部分，不改动核心分析内容
- **不删除信息**：所有原始分析内容都要保留，只是调整位置
- **总文章数**：88 → 86（-2），变化 2.3%，符合 ±10% 约束
- **风格**：保持原有专业教程风格，只去重不润色

## 1. 重复内容处理策略

### 1.1 完全合并策略（针对重度重复）

**适用场景**：两篇文章主题完全一致，内容重复度 >40%

**执行步骤：**

1. **打开主文章**（保留的那篇）
2. **识别主文章缺少的内容**：从待删除文章提取独有的内容段落
3. **插入到主文章合适位置**：按照逻辑顺序插入
4. **更新主文章目录/标题**：确保覆盖所有内容
5. **删除待合并文章**：更新 README 目录
6. **更新所有指向旧文章的链接**：改为指向主文章

**示例：** 合并 `18-compact` → `23-Token节约优化`

- 主文章 `23` 已有：三层优化体系、Session Memory、Microcompact、自动阈值计算
- 待合并 `18` 独有：`/compact` 命令实现、commands/compact/ 目录逐行分析、分组算法、base提示词
- 操作：在 `23` 增加一章「4. `/compact` 命令实现」，插入 `18` 的 2~3 节内容
- 删除 `18`，所有指向 `18` 的链接改为 `23`

### 1.2 导论去重策略（针对中度重复）

**适用场景**：深度文章重复了主文章的概述内容，需要保留深度解析，删除概述重复

**执行步骤：**

1. **在文章开头增加前置知识引用块**：
   ```markdown
   > **前置知识:** 阅读本文前，建议先阅读:
   > - [03-工具系统设计](./03-工具系统设计.md) - 理解Tool接口与buildTool工厂
   > - [07-权限与安全系统](./07-权限与安全系统.md) - 理解权限架构
   ```
2. **识别并删除重复段落**：
   - 删除重复的整体架构概述
   - 删除重复的基础概念定义（如 `PermissionMode`、`buildTool` 等）
   - 保留属于本文范围的特有内容
3. **调整章节结构**：确保删除后上下文连贯
4. **不保留空章节**：如果整个章节都是重复，整节删除

**规则**：只要概念已经在 L2 主文章讲过，L3 深度文章就必须引用，不能重复。

### 1.3 轻度概念重复策略（针对基础概念在多篇重复）

**适用场景**：`lazySchema`、`buildTool` 等基础概念在多篇工具详解中重复

**执行策略**：

- 保留一次完整讲解（在 L2 主文章 `03-工具系统设计`）
- 其他文章只留一句话说明 + 引用：
  ```markdown
  本文使用 `buildTool` 工厂函数创建工具，详见 [03-工具系统设计](./03-工具系统设计.md)
  ```
- 删除多行重复的代码分析

## 2. 结构优化策略

### 2.1 层级调整策略（12-26 从 L3 → L2）

**执行步骤：**

1. **README.md 更新**：修改 L2/L3 文章范围表
2. **文章文件无需移动位置**：只需要在目录中更新分类
3. **推荐阅读路径 更新**：按照新的 L1-L2-L3-L4 调整阅读顺序
4. **无需修改文章内容**：层级调整只是目录分类变化，文章内容不动

### 2.2 目录与导航优化

**每个文章开头必须保留阅读指引：**

```markdown
> **阅读指引:** 建议先阅读 [前文](./xx.md)，再阅读本文。具体xxx见 [另一文](./yy.md)
```

** Benefits:**

- 引导读者按逻辑顺序阅读
- 减少碎片化
- 自然体现层级关系

### 2.3 代码引用统一规范

所有代码引用必须：
- 标明 `文件路径:行数` 范围
- 代码片段保持原样，不修改
- 如果代码在多个文章重复引用，只需要保留引用，无需删除（代码引用不算内容重复）

## 3. 源码对齐策略

**规则：文章与源码冲突，以源码为准**

### 3.1 检查步骤

对于每篇文章：

1. **验证文件路径**：文章中提到的文件路径是否与实际源码一致
2. **验证行数**：文章标注的行数与源码是否一致
3. **验证代码片段**：代码片段是否与源码一致
4.**修复偏差**：如果有不一致，更正文件路径和行数

### 3.2 当前已发现的源码覆盖缺口

| 模块 | 当前状态 | 建议 |
|:-----|:---------|:------|
| Hook 钩子系统 | 只有 09 概述，无深度文章 | 在 L3 增加一篇 `Hook钩子系统详解.md`，不算新增（88→86，还有减少额度）|
| Plugin 插件系统 | 只有 09 概述，62 是开发指南，无深度解析 | 不需要新增，62 已涵盖开发，够了 |
| Remote 远程代理 | 只有 29 提到，无深度文章 | 不需要新增，属于高级功能，保持现状即可 |
| VSCode 扩展 | 无文章，只有 Chrome 扩展有文章 | 不需要新增，边缘功能 |

**结论**：不需要新增文章，当前覆盖已满足，缺口都是边缘功能，不影响体系完整性。

## 4. 术语统一策略

### 4.1 术语对照表统一到 91-概念术语表.md

  - `压缩` = `compact`，不混用 `浓缩`、`摘要`（`摘要` 是动作结果，`compact` 是过程）

### 4.2 链接统一规范

- 相对路径链接：`./filename.md` 格式正确
- 锚点链接：`#章节标题` 保持不变
- 所有链接必须指向 `.md` 扩展名，不带扩展名的添加扩展名
