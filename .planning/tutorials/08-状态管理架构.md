# Claude Code CLI 状态管理架构详解

**分析日期:** 2026-04-02

---

## 1. 概述

### 1.1 CLI状态管理的特殊性

Claude Code CLI采用**双轨状态管理架构**，这是CLI应用的独特设计：

**全局状态 (bootstrap/state.ts)**
- 模块级单例模式
- 跨进程生命周期
- 不依赖React渲染循环
- 用于：成本追踪、会话ID、模型设置、遥测数据

**React状态 (state/AppStateStore.ts)**
- 发布订阅模式 (Store<T>)
- 触发Ink组件重新渲染
- 用于：UI交互、任务列表、权限上下文

**核心区别：**
```typescript
// 全局状态 - 直接访问，无需订阅
const sessionId = getSessionId()  // bootstrap/state.ts

// React状态 - 订阅触发渲染
const appState = useAppState()    // React hook订阅Store变化
```

### 1.2 为什么需要双轨设计？

**性能考量：**
- 全局状态避免每次渲染都调用Date.now()
- 成本累积频繁更新（每个API调用）
- React状态批量更新机制不适合高频写入

**架构隔离：**
- bootstrap/state.ts是导入DAG的叶子节点
- 不能依赖src/utils/的任何模块
- AppState可以导入复杂类型（Tool、TaskState等）

**持久化差异：**
- 全局状态：会话恢复时从.jsonl文件重建
- React状态：进程启动时初始化，不持久化

```typescript
export function registerHookCallbacks(
  hooks: Partial<Record<HookEvent, RegisteredHookMatcher[]>>,
): void {
  if (!STATE.registeredHooks) {
    STATE.registeredHooks = {}
  }
  
  // 合而非覆盖（可能多次调用）
  for (const [event, matchers] of Object.entries(hooks)) {
    const eventKey = event as HookEvent
    if (!STATE.registeredHooks[eventKey]) {
      STATE.registeredHooks[eventKey] = []
    }
    STATE.registeredHooks[eventKey]!.push(...matchers)
  }
}

export function clearRegisteredPluginHooks(): void {
  if (!STATE.registeredHooks) return
  
  const filtered: Partial<Record<HookEvent, RegisteredHookMatcher[]>> = {}
  for (const [event, matchers] of Object.entries(STATE.registeredHooks)) {
    // 只保留callback hooks（没有pluginRoot的）
    const callbackHooks = matchers.filter(m => !('pluginRoot' in m))
    if (callbackHooks.length > 0) {
      filtered[event as HookEvent] = callbackHooks
    }
  }
  
  STATE.registeredHooks = Object.keys(filtered).length > 0 ? filtered : null
}
```

### 3.1 state/store.ts - Store<T>发布订阅

**核心类型定义：**
```typescript
type Listener = () => void
type OnChange<T> = (args: { newState: T; oldState: T }) => void

export type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}
```

**createStore工厂函数：**
```typescript
export function createStore<T>(
  initialState: T,
  onChange?: OnChange<T>,
): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()
  
  return {
    getState: () => state,
    
    setState: (updater: (prev: T) => T) => {
      const prev = state
      const next = updater(prev)
      
      // Object.is优化：相同引用则跳过通知
      if (Object.is(next, prev)) return
      
      state = next
      onChange?.({ newState: next, oldState: prev })
      
      // 触发所有监听器
      for (const listener of listeners) listener()
    },
    
    subscribe: (listener: Listener) => {
      listeners.add(listener)
      // 返回 unsubscribe 函数
      return () => listeners.delete(listener)
    },
  }
}
```

**设计要点：**

1. **Object.is优化：**
   - 状态未改变时跳过通知
   - 避免不必要的渲染

2. **Updater函数模式：**
   - `(prev: T) => T`确保不可变性
   - 强制创建新对象

3. **onChange回调：**
   - 状态变化时的副作用处理
   - 用于同步CCR、持久化等

4. **unsubscribe清理：**
   - 返回清理函数，避免内存泄漏
   - React组件卸载时调用

### 3.3 state/onChangeAppState.ts - 状态变更监听

**onChangeAppState函数：**
```typescript
export function onChangeAppState({
  newState,
  oldState,
}: {
  newState: AppState
  oldState: AppState
}) {
  // Permission mode同步到CCR
  const prevMode = oldState.toolPermissionContext.mode
  const newMode = newState.toolPermissionContext.mode
  if (prevMode !== newMode) {
    const prevExternal = toExternalPermissionMode(prevMode)
    const newExternal = toExternalPermissionMode(newMode)
    if (prevExternal !== newExternal) {
      notifySessionMetadataChanged({
        permission_mode: newExternal,
        is_ultraplan_mode: newExternal === 'plan' && newState.isUltraplanMode
          ? true : null,
      })
    }
    notifyPermissionModeChanged(newMode)
  }
  
  // mainLoopModel持久化
  if (newState.mainLoopModel !== oldState.mainLoopModel) {
    if (newState.mainLoopModel === null) {
      updateSettingsForSource('userSettings', { model: undefined })
      setMainLoopModelOverride(null)
    } else {
      updateSettingsForSource('userSettings', { model: newState.mainLoopModel })
      setMainLoopModelOverride(newState.mainLoopModel)
    }
  }
  
  // expandedView持久化
  if (newState.expandedView !== oldState.expandedView) {
    saveGlobalConfig(current => ({
      ...current,
      showExpandedTodos: newState.expandedView === 'tasks',
      showSpinnerTree: newState.expandedView === 'teammates',
    }))
  }
  
  // verbose持久化
  if (newState.verbose !== oldState.verbose) {
    saveGlobalConfig(current => ({ ...current, verbose: newState.verbose }))
  }
  
  // settings变化时清理认证缓存
  if (newState.settings !== oldState.settings) {
    clearApiKeyHelperCache()
    clearAwsCredentialsCache()
    clearGcpCredentialsCache()
    
    if (newState.settings.env !== oldState.settings.env) {
      applyConfigEnvironmentVariables()
    }
  }
}
```

**设计要点：**

1. **Permission Mode同步：**
   - 所有`setAppState`调用都会触发CCR同步
   - 不需要在各mutation path手动调用
   - `toExternalPermissionMode`转换内部模式名称

2. **Ultraplan标志：**
   - 只在第一次plan cycle设置
   - null使用RFC 7396语义（删除key）

3. **认证缓存清理：**
   - settings变化时自动清理
   - 确保新认证配置立即生效

```typescript
export const getGitStatus = memoize(async (): Promise<string | null> => {
  if (process.env.NODE_ENV === 'test') return null
  
  const startTime = Date.now()
  const isGit = await getIsGit()
  
  if (!isGit) return null
  
  try {
    const [branch, mainBranch, status, log, userName] = await Promise.all([
      getBranch(),
      getDefaultBranch(),
      execFileNoThrow(gitExe(), ['--no-optional-locks', 'status', '--short']),
      execFileNoThrow(gitExe(), ['--no-optional-locks', 'log', '--oneline', '-n', '5']),
      execFileNoThrow(gitExe(), ['config', 'user.name']),
    ])
    
    // 截断超长status
    const truncatedStatus =
      status.length > MAX_STATUS_CHARS
        ? status.substring(0, MAX_STATUS_CHARS) +
          '\n... (truncated because it exceeds 2k characters)'
        : status
    
    return [
      `This is the git status at the start of the conversation.`,
      `Current branch: ${branch}`,
      `Main branch: ${mainBranch}`,
      `Git user: ${userName}`,
      `Status:\n${truncatedStatus || '(clean)'}`,
      `Recent commits:\n${log}`,
    ].join('\n\n')
  } catch (error) {
    logError(error)
    return null
  }
})
```

**Git状态优化：**
- 使用`--no-optional-locks`避免锁竞争
- 并行执行多个git命令
- 截断超长输出（MAX_STATUS_CHARS = 2000）

**第一步：定义Store类型**
```typescript
// state/store.ts

type Listener = () => void

export type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: Listener) => () => void
}

export function createStore<T>(initialState: T): Store<T> {
  let state = initialState
  const listeners = new Set<Listener>()
  
  return {
    getState: () => state,
    setState: (updater) => {
      const next = updater(state)
      if (Object.is(next, state)) return
      state = next
      listeners.forEach(l => l())
    },
    subscribe: (listener) => {
      listeners.add(listener)
      return () => listeners.delete(listener)
    },
  }
}
```

## 6. 最佳实践总结

### 6.1 全局状态使用原则

**适用场景：**
- 成本、性能等高频更新数据
- 会话标识、路径等跨进程数据

- UI交互状态（用AppState）

## 7. 常见问题

### Q1: 为什么不用React Context管理全局状态？

**答：**
- Ink渲染周期与高频状态更新不匹配
- 成本累积每个API调用都更新，会触发过多渲染
- bootstrap/state.ts是DAG叶子，不能依赖React

### Q2: Object.is比较是否足够？

**答：**
- 对于不可变更新，Object.is足够
- AppState使用DeepImmutable强制不可变
- 如果直接修改对象，Object.is会失效（禁止）

### Q3: 为什么AppState不持久化？

**答：**
- UI状态在启动时重建更简单
- Tasks通过.jsonl逐条记录
- MCP连接需要重新建立（服务器可能重启）

### Q4: 如何避免循环依赖？

**答：**
- bootstrap/state.ts不导入src/utils/
- AppState导入复杂类型，但不导入utils实现
- 使用cachedClaudeMdContent打破classifier循环

### Q5: 如何处理会话切换？

**答：**
- 使用switchSession(sessionId, projectDir)
- 订阅onSessionSwitch回调
- 清理planSlugCache避免内存泄漏
