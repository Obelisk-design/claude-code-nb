## 1. 概述

### 1.1 命令系统用户界面

Claude Code CLI 提供丰富的斜杠命令（slash commands）系统，让用户能够快速执行各种操作：

**命令触发方式：**

1. **在对话中直接输入**
   ```
   /clear      # 清理当前会话
   /commit     # 创建Git提交
   /help       # 显示帮助
   ```

2. **命令行参数启动**
   ```bash
   claude --resume          # 恢复上次会话
   claude -r <session-id>   # 恢复指定会话
   claude --clear           # 启动时清理
   ```

3. **快捷键触发**
   - `Ctrl+L` - 清理屏幕
   - `Ctrl+B` - 后台任务
   - `/` + `Tab` - 命令自动补全

**命令类型：**

| 类型 | 说明 | 示例 |
|:---|:---|:---|
| `local` | 本地立即执行 | `/clear`, `/cost`, `/exit` |
| `local-jsx` | 打开UI面板 | `/resume`, `/config`, `/mcp` |
| `prompt` | 发送提示到AI | `/commit`, `/review`, `/init` |

## 2. 会话管理命令

### 2.1 /clear - 清理会话

**命令定义：** `commands/clear/index.ts`

**用户使用场景：**

1. **开始新任务** - 当前会话上下文混乱，想要全新开始
2. **释放上下文** - Token预算不足，需要清理历史消息
3. **切换项目** - 从一个项目切换到另一个，清除旧状态
4. **解决卡顿** - 会话状态异常，清理重置

**清理范围选择：**

`/clear` 命令会清理以下内容：

| 清理项 | 说明 | 是否保留 |
|:---|:---|:---:|
| 对话历史 | 所有消息记录 | ❌ 清空 |
| 文件缓存 | 读取过的文件内容 | ❌ 清空 |
| MCP连接 | 服务器连接状态 | ❌ 重置 |
| 会话ID | 当前会话标识 | ❌ 生成新ID |
| 后台任务 | 正在运行的后台任务 | ✅ 保留 |
| 成本统计 | 当前会话成本 | ❌ 重置 |
| Git状态 | 工作目录 | ✅ 保留 |

**操作步骤：**

1. 在对话中输入 `/clear`
2. 系统立即执行清理（无需确认）
3. 屏幕显示空白的全新会话界面
4. 可以开始新的对话

**注意事项：**

> ⚠️ **重要提示**
> 
> - `/clear` **不可撤销** - 清理后无法恢复对话历史
> - 后台任务会继续运行 - Ctrl+B启动的任务不受影响
> - 会话文件仍保存在磁盘 - 只是当前进程切换到新会话

**常见问题：**

**Q: `/clear` 后能否恢复之前的对话？**

A: 可以使用 `/resume` 恢复之前的会话。会话文件保存在：
- Linux/macOS: `~/.claude/projects/<project-hash>/`
- Windows: `%USERPROFILE%\.claude\projects\<project-hash>\`

**Q: 清理后文件修改是否丢失？**

A: `/clear` 只清理对话历史，不会影响文件系统。所有已写入的文件修改都会保留。

**Q: 清理会关闭MCP服务器吗？**

A: 会重置MCP连接状态，但服务器配置保留。下次使用工具时会重新连接。

### 2.2 /resume - 恢复会话

**命令定义：** `commands/resume/index.ts`, `commands/resume/resume.tsx`

**用户使用场景：**

1. **继续昨天的工作** - 恢复中断的开发任务
2. **查看历史决策** - 回顾之前的讨论和方案
3. **跨终端继续** - 在不同机器/终端继续工作
4. **多任务切换** - 在多个开发任务间切换

**如何查找历史会话：**

1. **不带参数运行 `/resume`**
   ```
   /resume
   ```
   系统显示会话选择器界面：

   ```
   ┌─ Resume Conversation ─────────────────────────────┐
   │  🔍 Search: [输入搜索词]                           │
   │                                                    │
   │  📁 Today                                          │
   │    ├─ "修复登录bug"              2小时前     45条消息 │
   │    ├─ "添加用户API"              3小时前     32条消息 │
   │    └─ (无标题)                   5小时前     18条消息 │
   │                                                    │
   │  📁 Yesterday                                      │
   │    ├─ "重构数据库层"              昨天 14:30  89条消息 │
   │    └─ "编写测试用例"              昨天 10:15  56条消息 │
   │                                                    │
   │  [Tab] 切换到所有项目  [Esc] 取消                   │
   └────────────────────────────────────────────────────┘
   ```

2. **使用会话ID恢复**
   ```
   /resume abc123-def456-ghi789
   ```
   会话ID可以从：
   - `/status` 命令显示
   - 会话文件名获取
   - 遥测日志查询

3. **使用会话标题搜索**
   ```
   /resume 修复登录bug
   ```
   精确匹配会话自定义标题

**恢复步骤：**

1. 输入 `/resume` 打开选择器
2. 使用 `↑` `↓` 键浏览会话列表
3. 按 `Enter` 选择恢复
4. 系统加载历史消息并显示

**跨项目恢复：**

当恢复不同项目的会话时，系统会：

```
┌─ 跨项目恢复 ─────────────────────────────────────┐
│                                                    │
│  此会话来自不同的目录。                            │
│                                                    │
│  要恢复，请运行：                                   │
│    claude -r abc123 --cwd /path/to/project        │
│                                                    │
│  (命令已复制到剪贴板)                              │
│                                                    │
│  [Enter] 关闭                                      │
   └────────────────────────────────────────────────┘
```

**最佳实践：**

1. **命名会话** - 使用 `/rename` 为会话设置有意义的标题
   ```
   /rename 修复登录bug - 2024-03-15
   ```

2. **定期清理** - 不要积累太多历史会话，定期清理不需要的

3. **使用worktree** - Git worktree用户可以在不同分支独立会话

4. **保存关键信息** - 重要决策应该记录到文档，不要仅依赖会话历史

**常见问题：**

**Q: 恢复会话后文件状态不一致怎么办？**

A: 恢复会话只加载对话历史，不会恢复文件状态。如果之前修改了文件：
- 使用 `/rewind` 回退到特定状态
- 或手动处理文件变更

**Q: 搜索不到想要的会话？**

A: 可能原因：
- 会话没有自定义标题（显示为"(无标题)"）
- 会话在不同项目目录下
- 会话是sidechain（后台任务会话）

尝试切换到"所有项目"视图：按 `Tab` 键

### 2.3 /branch - 会话分支

**命令定义：** `commands/branch/index.ts`, `commands/branch/branch.ts`

**用户使用场景：**

1. **探索不同方案** - 从某个点尝试多个解决方案
2. **保留决策点** - 在做出重大修改前创建备份点
3. **并行开发** - 同一功能的不同实现路径
4. **回滚备选** - 如果新方案失败，可以回到分支点

**分支创建场景：**

场景示例：你正在重构一个模块，想尝试两种方案

```
当前会话: "重构用户模块"
已完成: 分析现有代码、设计方案A
现在想: 同时尝试方案B

操作:
/branch 方案B尝试

系统输出:
✓ 已创建分支 "方案B尝试 (Branch)"
  当前在分支会话中
  原会话ID: abc123
  分支会话ID: def456
  
要恢复原会话: /resume abc123
```

**分支管理：**

创建分支后的状态：

| 项目 | 原会话 | 分支会话 |
|:---|:---|:---|
| 对话历史 | ✅ 保留 | ✅ 复制到分支点 |
| 后续消息 | ❌ 不包含分支后的消息 | ✅ 分支后的新消息 |
| 会话ID | 原ID不变 | 新UUID |
| 会话标题 | 原标题 | "原标题 (Branch)" |

**分支切换：**

在原会话和分支间切换：

```
# 从分支回到原会话
/resume <原会话ID>

# 从原会话进入分支
/resume <分支会话ID>
```

**分支命名规则：**

```
首次分支: "原标题 (Branch)"
再次分支: "原标题 (Branch 2)"
三次分支: "原标题 (Branch 3)"
...依此类推
```

**最佳实践：**

1. **关键决策前分支** - 在做出不可逆修改前创建分支点

2. **命名有意义的分支** - 使用参数命名
   ```
   /branch 尝试-缓存优化方案
   ```

3. **及时清理无用分支** - 确定方案后删除失败的分支会话文件

4. **不要过度分支** - 分支太多会造成管理混乱

### 2.4 /rewind - 回退操作

**命令定义：** `commands/rewind/index.ts`, `commands/rewind/rewind.ts`

**用户使用场景：**

1. **撤销错误修改** - Claude修改了不该改的文件
2. **回到稳定状态** - 当前状态有问题，回退到之前版本
3. **重新尝试方案** - 上一个方案不理想，回退重新尝试
4. **查看历史状态** - 查看某个时间点的代码状态

**回退触发条件：**

- Claude执行了错误的文件修改
- 代码修改导致测试失败
- 配置文件被错误更新
- 多个修改需要一起撤销

**操作步骤：**

1. 输入 `/rewind` 打开消息选择器

2. 界面显示历史消息列表：

   ```
   ┌─ Rewind to Message ───────────────────────────────┐
   │                                                    │
   │  选择回退点:                                        │
   │                                                    │
   │  [✓] 15:34 - Assistant: "已完成用户API重构"        │
   │       ├─ Edit: src/api/user.ts                    │
   │       ├─ Edit: src/api/auth.ts                    │
   │                                                    │
   │  [ ] 15:30 - Assistant: "开始重构"                 │
   │                                                    │
   │  [ ] 15:25 - User: "请重构用户API"                 │
   │                                                    │
   │  ↑↓ 选择  Enter 确认  Esc 取消                     │
   └────────────────────────────────────────────────────┘
   ```

3. 选择要回退的消息点

4. 系统执行回退：
   - 撤销该消息后的所有文件修改
   - 截断对话历史到该消息
   - 更新会话状态

**数据影响：**

回退操作会影响：

| 数据 | 影响 |
|:---|:---|
| 文件修改 | 撤销回退点之后的修改 |
| 对话历史 | 截断到回退点 |
| 文件历史 | 保留历史记录用于回退 |
| 成本统计 | 不回退（已消耗的成本保留） |

**注意事项：**

> ⚠️ **重要提示**
>
> - 回退依赖文件历史快照 - 如果没有快照，无法精确恢复
> - 回退只影响当前会话的修改 - 其他来源的修改不受影响
> - Git提交不会被撤销 - 需要手动 `git reset`
> - 回退后建议 `/commit` 保存状态

**常见问题：**

**Q: 回退失败怎么办？**

A: 可能原因：
- 文件历史快照未启用
- 文件被外部修改（IDE、其他工具）
- 修改跨越多个会话

解决方案：使用 Git 回退
```bash
git checkout HEAD -- <file>
git reset --hard <commit>
```

**Q: 回退后能否再前进？**

A: 不能。回退是永久操作。但可以用 `/branch` 在回退前创建分支点。

## 3. 代码操作命令

### 3.1 /commit - Git提交

**命令定义：** `commands/commit.ts`

**用户使用场景：**

1. **快速提交** - AI自动生成提交信息，无需手动编写
2. **批量提交** - 一次性提交多个相关修改
3. **规范化提交** - 按项目风格生成提交信息
4. **节省时间** - 减少手动git操作的繁琐

**自动生成提交信息：**

`/commit` 命令流程：

```
1. 分析当前git状态
   ├─ git status
   ├─ git diff HEAD
   ├─ git branch --show-current
   └─ git log --oneline -10

2. AI分析修改内容
   ├─ 判断修改类型: 新功能/修复/重构/文档
   ├─ 提取关键变更
   └─ 参考最近提交风格

3. 生成提交信息
   ├─ 简洁描述(1-2句)
   ├─ 关注"why"而非"what"
   └─ 匹配项目风格

4. 执行提交
   └─ git add + git commit
```

**提交前检查：**

系统自动检查：

| 检查项 | 说明 |
|:---|:---|
| 空提交 | 无修改时不创建提交 |
| 秘密文件 | 检测 `.env`, `credentials.*` 等敏感文件 |
| 提交风格 | 参考最近10条提交记录 |
| Git hooks | 不跳过 hooks（除非用户明确要求） |

**多文件提交：**

`/commit` 会一次性处理所有相关修改：

```
当前状态:
 modified:   src/api/user.ts
 modified:   src/api/auth.ts
 modified:   tests/user.test.ts
 new file:   docs/api-design.md

执行 /commit:
✓ 已分析所有修改
✓ 生成提交信息: "添加用户认证API，包含登录/注销功能"
✓ 已暂存所有文件
✓ 已创建提交 abc1234

1 file changed, 45 insertions(+), 12 deletions(-)
```

**最佳实践：**

1. **提交前review** - 使用 `/diff` 查看修改内容

2. **确保测试通过** - 提交前运行测试
   ```
   请运行测试后再提交
   ```

3. **不要提交秘密** - 系统会警告，但仍需用户注意

4. **清晰的对话上下文** - 让AI理解修改意图

5. **一次做一件事** - 避免混合不同目的的修改

**Git安全协议：**

系统遵循的安全规则：

- ❌ **永不**修改 git config
- ❌ **永不**跳过 hooks（除非用户明确要求）
- ❌ **永不**使用 `--amend`（除非用户明确要求）
- ❌ **永不**提交检测到的秘密文件
- ❌ **永不**创建空提交
- ❌ **永不**使用 `-i` 交互式命令

**常见问题：**

**Q: 提交信息不满意怎么办？**

A: 可以：
- 在对话中提供更多上下文
- 明确指定提交风格
- 手动 `git commit --amend` 修改

**Q: 如何提交部分文件？**

A: 先手动 `git add` 指定文件，再运行 `/commit`：
```bash
git add src/api/user.ts
# 然后在对话中运行 /commit
```

**Q: 检测到秘密文件警告？**

A: 系统会警告但不阻止。建议：
- 将秘密文件加入 `.gitignore`
- 使用环境变量替代硬编码
- 使用 `/config` 配置排除文件

### 3.2 /review - 代码审查

**命令定义：** `commands/review.ts`

**用户使用场景：**

1. **审查PR** - 快速审查Pull Request代码变更
2. **发现问题** - AI分析潜在bug、安全问题
3. **改进建议** - 获取代码改进建议
4. **学习代码** - 理解他人的代码设计

**审查范围选择：**

```
/review           # 显示PR列表供选择
/review 123       # 审查指定PR
/review main      # 审查当前分支与main的差异
```

**审查流程：**

```
1. 获取PR信息
   ├─ gh pr list (无参数时)
   ├─ gh pr view <number>
   └─ gh pr diff <number>

2. AI分析
   ├─ 理解PR目的
   ├─ 检查代码质量
   ├─ 分析潜在问题
   └─ 提供改进建议

3. 输出审查报告
   ├─ PR概述
   ├─ 代码质量分析
   ├─ 具体改进建议
   ├─ 潜在风险
   └─ 安全考虑
```

**审查结果解读：**

审查报告包含：

| 部分 | 说明 |
|:---|:---|
| PR概述 | 描述PR的目的和范围 |
| 代码质量 | 编码风格、可读性、结构 |
| 改进建议 | 具体的修改建议 |
| 潜在风险 | 可能的问题和bug |
| 安全考虑 | 安全相关问题 |
| 测试覆盖 | 测试充分性评估 |

**修复建议应用：**

审查后可以直接在对话中让Claude实施建议：

```
/review 123

[审查报告...]

请修复第3条建议的安全问题
✓ 已修复 input validation
✓ 已添加类型检查

请运行测试验证
✓ 测试通过
```

**最佳实践：**

1. **先看概述** - 理解PR目的再审查细节

2. **关注安全** - 特别注意安全问题

3. **逐步修复** - 不要一次性修改太多

4. **验证修改** - 每次修改后运行测试

5. **沟通讨论** - 在PR评论中讨论重大问题

**注意事项：**

> ⚠️ 使用 `/review` 需要 GitHub CLI (`gh`) 已安装并认证

**常见问题：**

**Q: 没有 `gh` 命令怎么办？**

A: 安装GitHub CLI：
```bash
# macOS
brew install gh

# Linux
sudo apt install gh  # Debian/Ubuntu
sudo dnf install gh  # Fedora

# 认证
gh auth login
```

**Q: 审查私有仓库PR？**

A: 需要 `gh` 认证且有仓库访问权限。

### 3.3 /diff - 差异查看

**命令定义：** `commands/diff/index.ts`

**用户使用场景：**

1. **查看未提交修改** - 了解当前修改内容
2. **确认变更** - 提交前确认修改符合预期
3. **定位问题** - 查看哪些文件被修改
4. **对比版本** - 查看不同版本间的差异

**查看范围：**

```
/diff             # 查看当前所有未提交修改
/diff HEAD~3      # 与3个提交前对比
/diff main        # 与main分支对比
```

**输出格式：**

`/diff` 显示统一的差异视图：

```
┌─ Diff View ──────────────────────────────────────┐
│                                                   │
│ 📁 src/api/user.ts                               │
│ ├─ 15 additions, 8 deletions                     │
│                                                   │
│ - function oldLogin() {                          │
│ + function newLogin(credentials: Credentials) {  │
│ +   validateInput(credentials);                  │
│ +   return authenticate(credentials);            │
│ -   return loginLegacy();                        │
│                                                   │
│ 📁 tests/user.test.ts                            │
│ ├─ 12 additions                                  │
│                                                   │
│ + test('newLogin validates input', () => {       │
│ +   expect(newLogin(null)).toThrow();            │
│ + });                                            │
│                                                   │
└───────────────────────────────────────────────────┘
```

**结果分析：**

差异视图可以帮助：

| 用途 | 说明 |

```json
    },

      "command": "mcp-server-github",
      "args": [],

      }
    }
  }
}
```

通过 `/config` 面板添加，或手动编辑配置。

**管理服务器：**

```
# 启用/禁用服务器
/mcp enable filesystem
/mcp disable filesystem
/mcp enable all
/mcp disable all

/mcp reconnect filesystem
```

常见问题：

| 问题 | 原因 | 解决 |

| 权限错误 | 环境变量缺失 | 配置env变量 |
| 工具不显示 | 初始化失败 | `/mcp reconnect` |

排查步骤：

1. 运行 `/doctor` 检查MCP状态
2. 检查服务器日志
3. 尝试 `/mcp reconnect`

**最佳实践：**

1. **按需启用** - 只启用需要的服务器减少开销

## 7. 常见问题解决

### 7.1 命令不响应

**症状：** 输入命令后无反应或卡住

**可能原因：**

| 原因 | 解决方法 |
|:---|:---|
| API请求进行中 | 等待或 `Ctrl+C` 取消 |
| MCP服务器连接超时 | `/mcp reconnect` 或禁用服务器 |
| 网络问题 | 检查网络连接 |
| 配置错误 | `/doctor` 诊断 |

**解决步骤：**

1. 等待几秒观察
2. `Ctrl+C` 取消当前操作
3. `/doctor` 诊断问题
4. 检查 `/status` 状态
5. 重启CLI

---

### 7.2 权限问题

**症状：** 工具执行被拒绝或沙箱错误

**常见错误：**

| 错误 | 原因 | 解决 |
|:---|:---|:---|
| `Permission denied` | 文件不在允许目录 | 添加目录到白名单 |
| `Sandbox violation` | 违反沙箱规则 | 检查沙箱配置 |
| `Tool not allowed` | 工具被禁用 | `/config` 启用工具 |
| `Bash blocked` | 命令被阻止 | 检查dangerous patterns |

**解决方法：**

1. 查看具体错误信息
2. 检查 `/config` 权限设置
3. 添加需要的目录到允许列表：
   ```
   /config
   > Permissions > Allowed directories
   > 添加 /path/to/needed/dir
   ```
4. 或临时禁用沙箱（不推荐）：
   ```
   /sandbox-toggle
   ```

---

### 7.3 配置问题

**症状：** 配置不生效或配置错误

**常见问题：**

| 问题 | 解决 |
|:---|:---|:---|
| 配置文件语法错误 | 检查JSON语法，用 `jq` 验证 |
| 配置不生效 | 检查优先级和层级 |
| MCP配置错误 | 检查command路径和env |
| Hook配置无效 | 检查事件类型和matcher |

**排查步骤：**

1. 运行 `/doctor` 检查配置
2. 手动检查配置文件语法：
   ```bash
   jq . ~/.claude/settings.json
   jq . .claude/settings.json
   ```
3. 检查配置层级覆盖
4. 查看CLI启动日志

**配置文件位置：**

- 全局: `~/.claude/settings.json`
- 项目: `.claude/settings.json`
- 本地: `.claude/settings.local.json`
- MCP: 同上 `mcpServers` 字段
- Hooks: 同上 `hooks` 字段
