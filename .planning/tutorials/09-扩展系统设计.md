# Claude Code CLI 扩展系统设计教程

**分析日期:** 2026-04-02

---

## 1. 概述

### 1.1 为什么CLI需要扩展系统

Claude Code CLI作为一款AI编程助手工具，其核心能力是通过模型调用工具完成代码操作。但实际使用场景远比基础工具调用复杂，扩展系统解决了以下关键问题：

**用户定制化需求**
- 不同团队有不同的工作流程和编码规范
- 用户需要复用成功的工作模式（如代码审查流程、部署流程）
- 企业需要集成内部工具和服务

**生命周期钩子**
- 工具执行前需要校验（PreToolUse）
- 工具执行后需要通知或记录（PostToolUse）
- 会话开始/结束需要初始化/清理（SessionStart/SessionEnd）

**能力边界扩展**
- 通过MCP（Model Context Protocol）连接外部数据源
- 通过Agent实现复杂的多步骤验证
- 通过HTTP Hook集成CI/CD系统

### 1.2 Skills/Plugins/Hooks三层架构

Claude Code CLI采用三层扩展架构，每层有不同的定位和用途：

```
┌─────────────────────────────────────────────────────┐
│                    Plugins (插件层)                   │
│  - 完整的功能包，可包含多个Skills/Hooks/MCP服务器       │
│  - 来自官方市场或第三方市场                            │
│  - 用户可通过 /plugin 命令启用/禁用                   │
│  - 文件位置: ~/.claude/plugins/                      │
├─────────────────────────────────────────────────────┤
│                    Skills (技能层)                    │
│  - 可复用的prompt模板，封装工作流程                    │
│  - 支持 /skill-name 命令直接调用                      │
│  - 来源: Bundled(内置)/User(用户)/Project(项目)/MCP   │
│  - 文件位置: .claude/skills/<name>/SKILL.md          │
├─────────────────────────────────────────────────────┤
│                    Hooks (钩子层)                     │
│  - 生命周期事件回调，拦截和增强操作                    │
│  - 类型: Command/Prompt/Agent/HTTP/Function         │
│  - 执行时机: Pre/Post工具调用，会话开始/结束等         │
│  - 配置位置: settings.json 或 plugin.json           │
└─────────────────────────────────────────────────────┘
```

**层级关系**

| 层级 | 包含关系 | 用户控制 | 持久化 |
|------|----------|----------|--------|
| Plugins | 可包含多个Skills和Hooks | `/plugin` UI控制 | 安装到plugins目录 |
| Skills | 独立prompt模板 | `/skill-name`调用 | SKILL.md文件 |
| Hooks | 事件回调 | settings.json配置 | settings.json或plugin.json |

### 2.1 skills/bundledSkills.ts 注册机制

**核心类型定义**

文件位置: `skills/bundledSkills.ts`

```typescript
// BundledSkillDefinition 定义了内置技能的完整结构
export type BundledSkillDefinition = {
  name: string                    // 技能名称，用于 /name 调用
  description: string             // 描述，显示在技能列表
  aliases?: string[]              // 别名列表
  whenToUse?: string              // 自动触发条件描述
  argumentHint?: string           // 参数提示，如 '[issue description]'
  allowedTools?: string[]         // 允许的工具列表，如 ['Read', 'Grep']
  model?: string                  // 指定模型
  disableModelInvocation?: boolean // 是否禁用自动调用
  userInvocable?: boolean         // 是否可用户手动调用
  isEnabled?: () => boolean       // 动态启用条件函数
  hooks?: HooksSettings           // 技能专属的钩子配置
  context?: 'inline' | 'fork'     // 执行上下文
  agent?: string                  // 关联的agent类型
  files?: Record<string, string>  // 内嵌参考文件
  getPromptForCommand: (          // Prompt生成函数
    args: string,
    context: ToolUseContext,
  ) => Promise<ContentBlockParam[]>
}
```

**注册流程**

```typescript
// 1. 注册函数实现
export function registerBundledSkill(definition: BundledSkillDefinition): void {
  const command: Command = {
    type: 'prompt',
    name: definition.name,
    description: definition.description,
    source: 'bundled',         // 标记为内置技能
    loadedFrom: 'bundled',
    // ... 其他属性映射
    getPromptForCommand: definition.getPromptForCommand,
  }
  bundledSkills.push(command)  // 添加到全局注册表
}

// 2. 获取所有已注册技能
export function getBundledSkills(): Command[] {
  return [...bundledSkills]  // 返回副本防止外部修改
}
```

**关键设计点**

1. **一次性写入注册表**: `bundledSkills`数组在模块初始化时填充，后续只读
2. **文件提取机制**: 如果skill定义了`files`字段，首次调用时会提取到临时目录
3. **安全写入**: 使用`O_NOFOLLOW | O_EXCL`防止符号链接攻击（参见`safeWriteFile`函数）

### 2.2 skills/loadSkillsDir.ts 加载逻辑

**文件来源优先级**

文件位置: `skills/loadSkillsDir.ts`

加载顺序（从高到低优先级）:
1. `policySettings` - 企业策略管理的技能
2. `userSettings` - 用户目录 `~/.claude/skills/`
3. `projectSettings` - 项目目录 `.claude/skills/`
4. `additionalDirs` - `--add-dir` 指定的额外目录
5. `commands_DEPRECATED` - 旧版 `/commands/` 目录

**加载函数核心逻辑**

```typescript
export const getSkillDirCommands = memoize(async (cwd: string): Promise<Command[]> => {
  // 1. 并行加载所有来源
  const [
    managedSkills,      // 策略管理
    userSkills,         // 用户目录
    projectSkillsNested, // 项目目录（多层级）
    additionalSkillsNested, // 额外目录
    legacyCommands,     // 旧版命令
  ] = await Promise.all([...])

  // 2. 合并并去重
  const allSkillsWithPaths = [
    ...managedSkills,
    ...userSkills,
    ...projectSkillsNested.flat(),
    ...additionalSkillsNested.flat(),
    ...legacyCommands,
  ]

  // 3. 基于realpath去重（处理符号链接）
  const fileIds = await Promise.all(
    allSkillsWithPaths.map(({ filePath }) => getFileIdentity(filePath))
  )

  // 4. 分离条件技能和常规技能
  for (const skill of deduplicatedSkills) {
    if (skill.paths && skill.paths.length > 0) {
      conditionalSkills.set(skill.name, skill)  // 存储条件技能
    } else {
      unconditionalSkills.push(skill)  // 直接返回
    }
  }

  return unconditionalSkills
})
```

```typescript
export async function discoverSkillDirsForPaths(
  filePaths: string[],
  cwd: string,
): Promise<string[]> {
  // 从文件路径向上遍历到cwd，发现.claude/skills目录
  while (currentDir.startsWith(resolvedCwd + pathSep)) {
    const skillDir = join(currentDir, '.claude', 'skills')
    if (!dynamicSkillDirs.has(skillDir)) {
      await fs.stat(skillDir)  // 检查目录是否存在
      if (!await isPathGitignored(currentDir, resolvedCwd)) {
        newDirs.push(skillDir)
      }
    }
    currentDir = dirname(currentDir)  // 向上一级
  }
  // 深度优先排序（更靠近文件的技能优先）
  return newDirs.sort((a, b) => b.split(pathSep).length - a.split(pathSep).length)
}
```

```typescript
// 内置插件定义类型
export type BuiltinPluginDefinition = {
  name: string                      // 插件名称
  description: string               // 描述
  version?: string                  // 版本
  skills?: BundledSkillDefinition[] // 提供的技能列表
  hooks?: HooksSettings             // 提供的钩子配置
  mcpServers?: Record<string, McpServerConfig> // MCP服务器配置
  isAvailable?: () => boolean       // 可用性检查函数
  defaultEnabled?: boolean          // 默认启用状态
}

// 注册函数
export function registerBuiltinPlugin(definition: BuiltinPluginDefinition): void {
  BUILTIN_PLUGINS.set(definition.name, definition)
}

// 获取内置插件（按启用状态分类）
export function getBuiltinPlugins(): { enabled: LoadedPlugin[], disabled: LoadedPlugin[] } {
  const settings = getSettings_DEPRECATED()
  for (const [name, definition] of BUILTIN_PLUGINS) {
    // 检查可用性
    if (definition.isAvailable && !definition.isAvailable()) continue

    // 用户设置 > 插件默认 > true
    const isEnabled = userSetting !== undefined
      ? userSetting === true
      : (definition.defaultEnabled ?? true)

    // 构建LoadedPlugin对象
    const plugin: LoadedPlugin = {
      name,
      manifest: { name, description, version },
      path: BUILTIN_MARKETPLACE_NAME,  // 特殊标记
      source: `${name}@builtin`,
      enabled: isEnabled,
      isBuiltin: true,
      hooksConfig: definition.hooks,
      mcpServers: definition.mcpServers,
    }
    // 分类
    if (isEnabled) enabled.push(plugin)
    else disabled.push(plugin)
  }
}
```

```typescript
// 格式: ~/.claude/plugins/cache/{marketplace}/{plugin}/{version}/
export function getVersionedCachePath(
  pluginId: string,  // 格式 "name@marketplace"
  version: string,   // semver或git SHA
): string {
  const { name: pluginName, marketplace } = parsePluginIdentifier(pluginId)
  // 安全化名称（防止路径遍历）
  const sanitizedMarketplace = marketplace.replace(/[^a-zA-Z0-9\-_]/g, '-')
  const sanitizedPlugin = pluginName.replace(/[^a-zA-Z0-9\-_]/g, '-')
  const sanitizedVersion = version.replace(/[^a-zA-Z0-9\-_.]/g, '-')
  return join(getPluginsDirectory(), 'cache', sanitizedMarketplace, sanitizedPlugin, sanitizedVersion)
}
```

```typescript
  hook: AgentHook,
  hookName: string,
  hookEvent: HookEvent,
  jsonInput: string,
  signal: AbortSignal,
  toolUseContext: ToolUseContext,

  // 1. 创建StructuredOutput工具
  const structuredOutputTool = createStructuredOutputTool()

  // 2. 准备工具列表（过滤掉禁止的工具）
  const tools: Tool[] = [
    ...filteredTools.filter(tool => !ALL_AGENT_DISALLOWED_TOOLS.has(tool.name)),
    structuredOutputTool,
  ]

  // 3. 构建系统prompt

    Use tools to inspect and verify.
    Return result with ${SYNTHETIC_OUTPUT_TOOL_NAME}: { ok: true/false, reason: ... }`

  // 4. 使用query()进行多轮执行
  for await (const message of query({
    messages: agentMessages,
    systemPrompt,
    tools,
    // ...
  })) {
    // 检查是否收到structured output
    if (message.attachment?.type === 'structured_output') {
      structuredOutputResult = parsed.data
      hookAbortController.abort()
    }

    }
  }

  if (!structuredOutputResult.ok) {
    return { outcome: 'blocking', blockingError: { blockingError: reason } }
  }
  return { outcome: 'success' }
```

```typescript
// 异步钩子执行不阻塞主流程
export function registerPendingAsyncHook({
  processId,
  hookId,
  asyncResponse,
  hookName,
  hookEvent,
  command,
  shellCommand,
}: {...}): void {
  const timeout = asyncResponse.asyncTimeout || 15000

  // 启动进度更新interval
  const stopProgressInterval = startHookProgressInterval({
    hookId,
    hookName,
    hookEvent,
    getOutput: async () => {
      const stdout = await shellCommand.taskOutput.getStdout()
      const stderr = shellCommand.taskOutput.getStderr()
      return { stdout, stderr, output: stdout + stderr }
    },
  })

  pendingHooks.set(processId, {
    processId,
    hookId,
    hookName,
    startTime: Date.now(),
    timeout,
    shellCommand,
    stopProgressInterval,
  })
}

// 检查异步钩子响应
export async function checkForAsyncHookResponses(): Promise<...> {
  for (const hook of pendingHooks.values()) {
    if (hook.shellCommand.status === 'completed') {
      const stdout = await hook.shellCommand.taskOutput.getStdout()
      // 解析JSON响应
      for (const line of stdout.split('\n')) {
        if (line.trim().startsWith('{')) {
          const parsed = jsonParse(line.trim())
          if (!('async' in parsed)) {
            response = parsed
          }
        }
      }
      await finalizeHook(hook, exitCode, outcome)
    }
  }
}
```

### 5.4 扩展点设计原则

**1. 单一职责**

每个扩展应专注于单一功能:
- Skill: 封装一个完整的工作流程
- Hook: 处理一个特定的生命周期事件
- Plugin: 组织相关的Skills和Hooks

**2. 明确的成功标准**

Skill中的每个步骤必须有`Success criteria`:
```markdown
### 1. Analyze Target
**Success criteria**: File loaded, structure identified, dependencies mapped.
```

**3. 安全的权限声明**

只声明必要的工具权限:
```yaml
allowed-tools:
  - Bash(git:*)  # 仅git相关命令
  - Read         # 仅读取
```

**4. 条件过滤优化**

使用`if`字段避免不必要的Hook执行:
```json
{
  "type": "command",
  "command": "/path/to/hook.sh",
  "if": "Bash(git checkout*)"
}
```

**5. 错误处理约定**

Hook exit code约定:
| Exit Code | 含义 | 行为 |
|-----------|------|------|
| 0 | 成功 | 正常继续 |
| 2 | 阻塞错误 | 显示错误，停止/询问 |
| 其他 | 非阻塞错误 | 显示警告，继续 |

**6. 版本锁定**

对于关键Plugin，使用Git SHA锁定版本:
```json
{
  "enabledPlugins": {
    "critical-plugin@marketplace": {
      "enabled": true,
      "sha": "abc123def456..."
    }
  }
}
```

**7. 企业策略兼容**

检查`isAvailable()`函数:
```typescript
// Plugin定义
registerBuiltinPlugin({
  name: 'enterprise-plugin',
  isAvailable: () => {
    // 检查企业环境
    return process.env.ENTERPRISE_MODE === 'true'
  },
  // ...
})
```

**8. 测试覆盖**

为每个扩展编写测试:
- Skill: 测试prompt生成和参数替换
- Hook: 测试各种exit code场景
- Plugin: 测试加载和组件发现
