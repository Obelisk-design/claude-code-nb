# 批量操作与并行处理

**分析日期:** 2026-04-02

## 一、并行处理概述

### 1.1 核心设计理念

Claude Code CLI 在批量操作和并行处理方面采用了**分层并发策略**：

- **本地服务器（stdio/sdk）**：低并发（默认3），避免进程资源竞争
- **远程服务器（HTTP/SSE）**：高并发（默认20），充分利用网络连接
- **文件系统操作**：`Promise.all` 并行执行，但使用锁机制防止竞争
- **顺序敏感操作**：`sequential()` 包装器确保串行执行

### 1.2 并行处理的三种模式

| 模式 | 适用场景 | 工具 | 示例 |
|------|---------|------|------|
| **并发池** | 大量相似任务，需限制并发数 | `p-map` | MCP服务器连接 |
| **全并行** | 少量独立任务，无资源竞争 | `Promise.all` | Git状态获取 |
| **顺序执行** | 顺序敏感操作，避免竞态 | `sequential()` | 文件写入 |

**设计亮点：**

1. **动态并发槽位释放**：
   ```typescript
   // 注释说明（第2212-2217行）：
   // "Replaced 2026-03: previous implementation ran fixed-size sequential batches
   // (await batch 1 fully, then start batch 2). That meant one slow server in
   // batch N held up ALL servers in batch N+1, even if the other 19 slots were
   // idle. pMap frees each slot as soon as its server completes, so a single
   // slow server only occupies one slot instead of blocking an entire batch
   // boundary."

   // 旧批处理问题：固定批次边界导致整体阻塞
   // 新方案优势：单个槽位即时释放，避免级联阻塞
   ```

2. **分组并发策略**（第2388-2402行）：

```typescript
// 核心并发策略：分组并行
await Promise.all([
  // 本地服务器：低并发（默认3）
  // 原因：进程spawn有资源竞争，避免系统过载
  processBatched(
    localServers,
    getMcpServerConnectionBatchSize(), // 默认3
    processServer,
  ),

  // 远程服务器：高并发（默认20）
  // 原因：仅网络连接，无本地资源竞争
  processBatched(
    remoteServers,
    getRemoteMcpServerConnectionBatchSize(), // 默认20
    processServer,
  ),
])
```

**关键思考：为什么分组？**

- 本地服务器（stdio/sdk）需要spawn子进程
- 进程创建有系统开销（文件描述符、内存、调度）
- 远程服务器仅建立网络连接
- 网络连接成本低，可大幅提高并发
- **分组后每组独立并发池，互不干扰**

### 3.2 Promise.allSettled容错模式

**适用场景：** 需要收集所有结果（包括失败），不能因单个失败中断

#### 文件历史恢复（utils/fileHistory.ts 第963-1035行）

```typescript
// 批量恢复备份文件
await Promise.allSettled(
  fileHistorySnapshots.map(async snapshot => {
    const backupEntries = Object.values(snapshot.trackedFileBackups).filter(
      (backup): backup is typeof backup & { backupFileName: string } =>
        backup.backupFileName !== null,
    )

    // 每个snapshot内部也并行处理
    const results = await Promise.allSettled(
      backupEntries.map(async ({ backupFileName }) => {
        const oldBackupPath = resolveBackupPath(backupFileName, previousSessionId)
        const newBackupPath = join(newBackupDir, backupFileName)

        try {
          // 优先尝试硬链接（节省空间）
          await link(oldBackupPath, newBackupPath)
        } catch (e: unknown) {
          const code = getErrnoCode(e)
          if (code === 'EEXIST') {
            // 已存在：跳过（幂等性）
            return
          }
          if (code === 'ENOENT') {
            // 源文件不存在：记录错误，继续其他文件
            logError(new Error(`FileHistory: Backup file does not exist in ${previousSessionId}`))
            throw e
          }
          // 硬链接失败：降级为文件复制
          try {
            await copyFile(oldBackupPath, newBackupPath)
          } catch (copyErr) {
            logError(new Error(`FileHistory: Error copying backup from previous session`))
            throw copyErr
          }
        }
      }),
    )

    // 统计失败数量（但不中断流程）
    const copyFailed = results.some(r => r.status === 'rejected')
    if (!copyFailed) {
      // 成功才记录snapshot
      void recordFileHistorySnapshot(snapshot.messageId, snapshot, false)
    } else {
      failedSnapshots++
    }
  }),
)

// 最后统一报告失败（而非立即抛出）
if (failedSnapshots > 0) {
  logEvent('tengu_file_history_resume_copy_failed', {
    numSnapshots: fileHistorySnapshots.length,
    failedSnapshots,
  })
}
```

**设计精髓：**

1. **两层Promise.allSettled**：
   - 外层：处理所有snapshot
   - 内层：处理snapshot内所有backup文件

2. **容错策略**：
   - 单文件失败不影响snapshot
   - 单snapshot失败不影响整体
   - 最终统计失败数量，统一报告

3. **硬链接优先**（第979行）：
   ```typescript
   await link(oldBackupPath, newBackupPath)
   // 优势：节省磁盘空间（同一文件的引用）
   // 降级：link失败时copyFile（跨文件系统等）
   ```

### 4.2 文件锁机制防止竞争

#### 任务创建的锁保护（utils/tasks.ts 第284-308行）

```typescript
export async function createTask(
  taskListId: string,
  taskData: Omit<Task, 'id'>,
): Promise<string> {
  const lockPath = await ensureTaskListLockFile(taskListId)

  let release: (() => Promise<void>) | undefined
  try {
    // 关键：获取排他锁
    release = await lockfile.lock(lockPath, LOCK_OPTIONS)

    // 锁内执行：读取最高ID + 分配新ID + 写入文件
    const highestId = await findHighestTaskId(taskListId)
    const id = String(highestId + 1)
    const task: Task = { id, ...taskData }
    const path = getTaskPath(taskListId, id)
    await writeFile(path, jsonStringify(task, null, 2))

    notifyTasksUpdated()
    return id
  } finally {
    // 关键：finally确保锁释放（即使异常）
    if (release) {
      await release()
    }
  }
}
```

**锁配置详解（第98-108行）：**

```typescript
const LOCK_OPTIONS = {
  retries: {
    retries: 30,        // 重试次数（高并发场景）
    minTimeout: 5,      // 最小等待5ms
    maxTimeout: 100,    // 最大等待100ms
  },
}

// 注释说明（第94-100行）：
// "Lock options: retry with backoff so concurrent callers (multiple Claudes
// in a swarm) wait for the lock instead of failing immediately. The sync
// lockSync API blocked the event loop; the async API needs explicit retries
// to achieve the same serialization semantics.
//
// Budget sized for ~10+ concurrent swarm agents: each critical section does
// readdir + N×readFile + writeFile (~50-100ms on slow disks), so the last
// caller in a 10-way race needs ~900ms. retries=30 gives ~2.6s total wait."

// 关键思考：
// - 10个并发agent竞争同一任务列表
// - 每个临界区耗时50-100ms（文件IO）
// - 最后一个竞争者需等待前9个完成
// - 总等待时间 ≈ 900ms
// - retries=30提供2.6s等待窗口（足够覆盖）
```

**任务认领的锁策略（第541-612行）：**

```typescript
export async function claimTask(
  taskListId: string,
  taskId: string,
  claimantAgentId: string,
  options: ClaimTaskOptions = {},
): Promise<ClaimTaskResult> {
  const taskPath = getTaskPath(taskListId, taskId)

  // 先检查任务是否存在（避免锁不存在文件）
  const taskBeforeLock = await getTask(taskListId, taskId)
  if (!taskBeforeLock) {
    return { success: false, reason: 'task_not_found' }
  }

  // 两种锁策略：
  if (options.checkAgentBusy) {
    // 策略1：列表级锁（原子检查agent是否忙碌）
    return claimTaskWithBusyCheck(taskListId, taskId, claimantAgentId)
  }

  // 策略2：任务级锁（仅锁单个任务文件）
  let release: (() => Promise<void>) | undefined
  try {
    release = await lockfile.lock(taskPath, LOCK_OPTIONS)

    // 锁内执行：
    // 1. 重新读取任务（防止TOCTOU）
    const task = await getTask(taskListId, taskId)
    if (!task) {
      return { success: false, reason: 'task_not_found' }
    }

    // 2. 检查是否已被其他agent认领
    if (task.owner && task.owner !== claimantAgentId) {
      return { success: false, reason: 'already_claimed', task }
    }

    // 3. 检查是否已完成
    if (task.status === 'completed') {
      return { success: false, reason: 'already_resolved', task }
    }

    // 4. 检查是否被阻塞（依赖未完成）
    const allTasks = await listTasks(taskListId)
    const unresolvedTaskIds = new Set(
      allTasks.filter(t => t.status !== 'completed').map(t => t.id),
    )
    const blockedByTasks = task.blockedBy.filter(id => unresolvedTaskIds.has(id))
    if (blockedByTasks.length > 0) {
      return { success: false, reason: 'blocked', task, blockedByTasks }
    }

    // 5. 认领任务（更新owner字段）
    const updated = await updateTaskUnsafe(taskListId, taskId, {
      owner: claimantAgentId,
    })
    return { success: true, task: updated! }
  } finally {
    if (release) {
      await release()
    }
  }
}
```

**TOCTOU防御机制：**

```typescript
// TOCTOU = Time-Of-Check-Time-Of-Use
// 问题：检查状态和使用状态之间存在时间窗口，可能被其他进程修改

// 防御策略：
// 1. 锁内重新读取：防止检查后状态变化
// 2. 检查后立即更新：最小化临界区
// 3. 列表级锁：原子检查多个条件

// 示例：claimTaskWithBusyCheck（第618-692行）
async function claimTaskWithBusyCheck(
  taskListId: string,
  taskId: string,
  claimantAgentId: string,
): Promise<ClaimTaskResult> {
  const lockPath = await ensureTaskListLockFile(taskListId)

  let release: (() => Promise<void>) | undefined
  try {
    // 列表级锁：原子读取所有任务
    release = await lockfile.lock(lockPath, LOCK_OPTIONS)

    const allTasks = await listTasks(taskListId)

    // 原子检查多个条件：
    // 1. 任务是否存在
    // 2. 是否已被认领
    // 3. 是否已完成
    // 4. 是否被阻塞
    // 5. agent是否忙碌（关键：检查其他任务）

    const task = allTasks.find(t => t.id === taskId)
    if (!task) {
      return { success: false, reason: 'task_not_found' }
    }

    if (task.owner && task.owner !== claimantAgentId) {
      return { success: false, reason: 'already_claimed', task }
    }

    if (task.status === 'completed') {
      return { success: false, reason: 'already_resolved', task }
    }

    const unresolvedTaskIds = new Set(
      allTasks.filter(t => t.status !== 'completed').map(t => t.id),
    )
    const blockedByTasks = task.blockedBy.filter(id => unresolvedTaskIds.has(id))
    if (blockedByTasks.length > 0) {
      return { success: false, reason: 'blocked', task, blockedByTasks }
    }

    // 关键：原子检查agent是否忙碌（防止agent认领多个任务）
    const agentOpenTasks = allTasks.filter(
      t =>
        t.status !== 'completed' &&
        t.owner === claimantAgentId &&
        t.id !== taskId,
    )
    if (agentOpenTasks.length > 0) {
      return {
        success: false,
        reason: 'agent_busy',
        task,
        busyWithTasks: agentOpenTasks.map(t => t.id),
      }
    }

    // 所有检查通过：认领任务
    const updated = await updateTask(taskListId, taskId, {
      owner: claimantAgentId,
    })
    return { success: true, task: updated! }
  } finally {
    if (release) {
      await release()
    }
  }
}
```

## 五、性能优化策略

### 5.1 并发数选择策略

| 操作类型 | 默认并发 | 优化考虑 | 可调范围 |
|---------|---------|---------|---------|
| **本地进程spawn** | 3 | 进程创建开销、文件描述符限制 | 1-10 |
| **网络连接** | 20 | 网络带宽、连接池大小 | 5-50 |
| **文件读取** | 无限制 | IO带宽、内存占用 | 根据文件大小 |
| **文件写入** | 串行化 | 磁盘IO竞争、数据一致性 | 必须串行 |

### 5.2 关键优化技巧

#### 技巧1：mtime优先检查（避免内容读取）

```typescript
// utils/fileHistory.ts 第600-634行
export async function checkOriginFileChanged(
  originalFile: string,
  backupFileName: string,
  originalStatsHint?: Stats,
): Promise<boolean> {
  // 优化1：优先使用hint避免重复stat
  let originalStats: Stats | null = originalStatsHint ?? null
  if (!originalStats) {
    try {
      originalStats = await stat(originalFile)
    } catch (e: unknown) {
      if (!isENOENT(e)) return true
    }
  }

  let backupStats: Stats | null = null
  try {
    backupStats = await stat(backupPath)
  } catch (e: unknown) {
    if (!isENOENT(e)) return true
  }

  // 优化2：mtime比较（快速路径）
  if (originalStats.mtimeMs < backupStats.mtimeMs) {
    return false  // 文件未修改，无需内容比较
  }

  // 优化3：内容比较（慢速路径，仅在必要时）
  const [originalContent, backupContent] = await Promise.all([
    readFile(originalFile, 'utf-8'),
    readFile(backupPath, 'utf-8'),
  ])
  return originalContent !== backupContent
}
```

```typescript
} catch (e: unknown) {
```

```typescript
/**
 * 简易并发池实现
 * 核心思路：维护任务队列 + 活动计数器
 */
class ConcurrencyPool {
  private activeCount = 0
  private queue: Array<{
    task: () => Promise<void>
    resolve: () => void
  }> = []

  constructor(private maxConcurrency: number) {}

  async run<T>(task: () => Promise<T>): Promise<T> {
    // 如果达到并发上限，加入队列等待
    if (this.activeCount >= this.maxConcurrency) {
      await new Promise<void>(resolve => {
        this.queue.push({ task: task as () => Promise<void>, resolve })
      })
    }

    this.activeCount++

    try {
      const result = await task()
      return result
    } finally {
      this.activeCount--

      // 完成一个任务后，启动队列中的下一个
      if (this.queue.length > 0 && this.activeCount < this.maxConcurrency) {
        const next = this.queue.shift()!
        next.resolve()  // 解除等待，允许执行
        this.run(next.task)  // 启动任务（不等待结果）
      }
    }
  }
}

// 使用示例
const pool = new ConcurrencyPool(3)
await Promise.all(
  tasks.map(task => pool.run(() => processTask(task)))
)
// 最多同时执行3个任务
```

### 6.2 完整批量处理器

```typescript
/**
 * 完整批量处理器
 * 特性：
 * - 并发控制
 * - 错误收集
 * - 进度回调
 * - 可取消
 */
export class BatchProcessor<T, R> {
  private results: Array<{ status: 'fulfilled' | 'rejected'; value?: R; reason?: Error }> = []
  private completed = 0
  private cancelled = false

  constructor(
    private items: T[],
    private processor: (item: T) => Promise<R>,
    private concurrency: number,
    private onProgress?: (completed: number, total: number) => void,
  ) {}

  async run(): Promise<{
    successful: R[]
    failed: Array<{ item: T; error: Error }>
  }> {
    const pool = new ConcurrencyPool(this.concurrency)

    await Promise.all(
      this.items.map(async (item, index) => {
        if (this.cancelled) return

        try {
          const result = await pool.run(() => this.processor(item))
          this.results[index] = { status: 'fulfilled', value: result }
        } catch (error) {
          this.results[index] = { status: 'rejected', reason: error as Error }
        }

        this.completed++
        this.onProgress?.(this.completed, this.items.length)
      }),
    )

    return {
      successful: this.results
        .filter(r => r.status === 'fulfilled')
        .map(r => r.value!),
      failed: this.results
        .filter(r => r.status === 'rejected')
        .map((r, i) => ({ item: this.items[i], error: r.reason! })),
    }
  }

  cancel(): void {
    this.cancelled = true
  }
}

// 使用示例
const processor = new BatchProcessor(
  files,
  async file => {
    const content = await readFile(file, 'utf-8')
    return processContent(content)
  },
  5,  // 并发数
  (completed, total) => console.log(`${completed}/${total} files processed`),
)

const { successful, failed } = await processor.run()
console.log(`Success: ${successful.length}, Failed: ${failed.length}`)
```

### 6.3 sequential()实现解析

**文件位置：** `utils/sequential.ts`

```typescript
/**
 * 顺序执行包装器
 * 核心思路：队列 + 单处理循环
 */
export function sequential<T extends unknown[], R>(
  fn: (...args: T) => Promise<R>,
): (...args: T) => Promise<R> {
  const queue: QueueItem<T, R>[] = []
  let processing = false

  async function processQueue(): Promise<void> {
    // 关键1：防止重入
    if (processing) return
    if (queue.length === 0) return

    processing = true

    // 关键2：顺序处理队列
    while (queue.length > 0) {
      const { args, resolve, reject, context } = queue.shift()!

      try {
        const result = await fn.apply(context, args)
        resolve(result)  // 正确resolve对应的Promise
      } catch (error) {
        reject(error)    // 正确reject对应的Promise
      }
    }

    processing = false

    // 关键3：处理期间新加入的任务
    if (queue.length > 0) {
      void processQueue()  // 不等待，避免递归栈溢出
    }
  }

  return function (this: unknown, ...args: T): Promise<R> {
    // 每次调用返回新Promise
    return new Promise((resolve, reject) => {
      queue.push({ args, resolve, reject, context: this })
      void processQueue()  // 触发处理（如果未在处理中）
    })
  }
}

// 使用示例
const sequentialWrite = sequential(async (filePath: string, content: string) => {
  await writeFile(filePath, content)
})

// 多个并发调用会自动排队
await Promise.all([
  sequentialWrite('file1.txt', 'content1'),  // 第1个执行
  sequentialWrite('file2.txt', 'content2'),  // 等第1个完成
  sequentialWrite('file3.txt', 'content3'),  // 等第2个完成
])
```

**设计精髓：**

1. **队列+单循环**：避免复杂锁机制
2. **Promise对应**：每个调用返回独立Promise
3. **防重入**：processing标志防止并发processQueue
4. **延迟处理**：process期间新任务自动触发下一轮

### 6.4 文件锁实现（简化版）

```typescript
/**
 * 简易文件锁
 * 核心思路：原子创建锁文件 + 重试等待
 */
class FileLock {
  private lockPath: string

  constructor(private filePath: string) {
    this.lockPath = filePath + '.lock'
  }

  async acquire(options: {
    retries: number
    minTimeout: number
    maxTimeout: number
  }): Promise<() => Promise<void>> {
    for (let attempt = 0; attempt < options.retries; attempt++) {
      try {
        // 原子创建：wx标志确保文件不存在时才成功
        await writeFile(this.lockPath, String(process.pid), { flag: 'wx' })
        // 成功：返回释放函数
        return async () => {
          await unlink(this.lockPath)
        }
      } catch (error) {
        if (getErrnoCode(error) !== 'EEXIST') {
          throw error  // 不是锁竞争，抛出错误
        }
        // 锁竞争：等待后重试
        const timeout = Math.min(
          options.minTimeout * Math.pow(2, attempt),
          options.maxTimeout,
        )
        await new Promise(resolve => setTimeout(resolve, timeout))
      }
    }

    throw new Error(`Failed to acquire lock after ${options.retries} retries`)
  }
}

// 使用示例
const lock = new FileLock('shared-resource.json')
const release = await lock.acquire({
  retries: 30,
  minTimeout: 5,
  maxTimeout: 100,
})

try {
  // 临界区操作
  const data = await readFile('shared-resource.json', 'utf-8')
  await writeFile('shared-resource.json', updateData(data))
} finally {
  await release()  // 必须释放锁
}
```

## 八、总结与最佳实践

### 8.1 并发模式选择指南

| 场景特征 | 推荐模式 | 示例 |
|---------|---------|------|
| **大量相似任务，有资源限制** | p-map并发池 | MCP服务器连接 |
| **少量独立任务，无竞争** | Promise.all | Git状态获取 |
| **顺序敏感操作** | sequential() | 文件写入 |
| **需收集所有结果（含失败）** | Promise.allSettled | 文件恢复 |
| **多条件原子检查** | 列表级锁 | 任务认领 |

### 8.2 性能优化 checklist

- [ ] **并发数调优**：根据资源类型选择合理并发
- [ ] **mtime优化**：快速路径避免不必要的IO
- [ ] **硬链接优先**：节省磁盘空间和时间
- [ ] **批量预处理**：先检查存在性再处理
- [ ] **并发预取**：提前启动不阻塞主流程
- [ ] **错误内吞**：单任务失败不影响整体
- [ ] **锁释放保障**：finally确保锁释放

### 8.3 常见陷阱提醒

1. ❌ `.map(async)`不等待 → 使用Promise.all/p-map
2. ❌ `.forEach(async)`不处理Promise → 改用for-await-of
3. ❌ 串行await退化 → 改为Promise.all并行
4. ❌ 无并发限制 → 使用p-map控制并发
5. ❌ 锁未释放 → 使用try-finally保障
